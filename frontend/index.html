How It Works

Students scan a rotating QR code or enter a manual code, confirm their location, confirm their device, and check in — all in under 30 seconds.

EduCore Flow — 7-Layer Security Stack

Step-by-step check-in process

01

Master List gate

Only students whose ID appears on the institution's Master List can register and access the system.

02

Identity — Profile Discovery

Student profile is loaded and verified. Account must be email-verified and active.

03

Device — Hardware UUID binding

The student's device UUID is checked. Attendance from an unregistered device is rejected.

04

Location — Haversine Geofence

EduLink verifies the student is inside the lecture hall radius before proceeding.

05

Presence — Dynamic QR / Manual Code

The student scans the rotating QR or enters the 6-character manual code on the projector.

06

Knowledge — Password confirmation

After scanning, the student confirms their EduCore password. Shared codes are rendered useless.

07

Integrity — SATE token generated

A SHA-256 SATE hash is computed over all six factors and stored on the record. Tamper-detection is built in.

v5 Feature

Lecturer Manual Override — Controlled & Audited

When a student fails to check in after the configured number of attempts, EduLink submits an override request — not a bypass. The lecturer sees everything before deciding.

Each request contains the student's profile, geofence result with distance, attempt count, QR token validity, and exact timestamp.

Approved records are stored with verification_method = 'manual_override' and a full immutable audit trail — who approved, when, from which device.

Security Architecture — v5

Seven independent proofs.
One attendance record.

Every valid EduCore attendance record must satisfy all seven layers simultaneously. Removing any one makes the record invalid or cryptographically detectable.

01 / Authorization

Master List Gate

Only students whose index number is on the institution's Master List can register and access the system.

02 / Identity

Profile Verification

Student account validated and email-verified. Profile is confirmed on every check-in against the enrolled list.

03 / Device

Hardware UUID Binding

Student's hardware UUID is bound on first login. Attendance from any other device is rejected outright.

04 / Location

Haversine Geofence

Geofence confirms physical presence inside the lecture hall. Students outside the radius cannot check in.

05 / Presence

Dynamic QR / Code

Codes rotate every 15 seconds and expire immediately after first use. Shared codes are automatically invalid.

06 / Knowledge

Password Confirmation

After scanning, the student confirms their EduCore password. Even a shared QR code is useless without it.

07 / Integrity

SATE Cryptographic Hash

SHA-256 token stored on every record. Hash covers student_id, session_id, qr_token, GPS, device_uuid, and timestamp.

SHA256(student_id | session_id | qr_token | lat | lng | device_uuid | timestamp)

What prevents a student from sharing their QR token?
EduCore v5 uses seven locks: Master List verification, identity confirmation, device UUID binding, GPS geofence, dynamic QR (rotates every 15s), password confirmation, and a SATE SHA-256 integrity hash. A student can share their QR code, but it expires in seconds, only works from their bound device, only works inside the geofence, and still requires their password.
What is Device Binding and how does it work?
On a student's first login, EduCore captures and stores a hardware UUID for their device. Every subsequent check-in validates against that stored UUID. If a student changes devices, the lecturer can reset the binding from the web terminal — the action is logged in the audit trail.
What is SATE and what does it prove?
SATE (Secure Attendance Token Engine) generates a SHA-256 hash from six inputs: student ID, session ID, QR token, GPS latitude, GPS longitude, device UUID, and timestamp. Any tamper with any of those fields produces a different hash — equivalent to a digital signature on the record.
Is the Lecturer Override a security loophole?
No — it's a controlled, audited exception. Override requests still require the student to have a valid QR token and be inside the geofence. Every approval is written to the immutable audit log with actor, device, and timestamp. It cannot be abused silently.
Who owns the attendance data?
The educator does. EduCore is built on an educator-owned data model — every record, class, and student is linked to the lecturer's account. Data is not shared, sold, or used for any purpose outside the platform without explicit consent.
